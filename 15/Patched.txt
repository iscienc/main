Step 2 — ICT_TradeManager.mqh: the guard function
Place directly ABOVE PreTradeChecks():

//+------------------------------------------------------------------+
//| True if a live position OR pending order already exists for this  |
//| SM chain (matched by magic + "#<chainId>" in the comment).        |
//+------------------------------------------------------------------+
bool HasOrderForChain(int chainId)
  {
   if(chainId <= 0) return false;
   string tag = "#" + IntegerToString(chainId);

   for(int i = PositionsTotal()-1; i >= 0; i--)
     {
      ulong t = PositionGetTicket(i);
      if(t == 0) continue;
      if(PositionGetString(POSITION_SYMBOL)  != _Symbol)       continue;
      if(PositionGetInteger(POSITION_MAGIC)  != InpMagicNumber) continue;
      if(StringFind(PositionGetString(POSITION_COMMENT), tag) >= 0) return true;
     }
   for(int i = OrdersTotal()-1; i >= 0; i--)
     {
      ulong t = OrderGetTicket(i);
      if(t == 0) continue;
      if(OrderGetString(ORDER_SYMBOL)  != _Symbol)       continue;
      if(OrderGetInteger(ORDER_MAGIC)  != InpMagicNumber) continue;
      if(StringFind(OrderGetString(ORDER_COMMENT), tag) >= 0) return true;
     }
   return false;
  }
Step 3 — ICT_TradeManager.mqh: enforce in PreTradeChecks()
Add early in PreTradeChecks() (after the terminal/EA/account checks, before spread/session):

   if(HasOrderForChain(g_currentSignal.chainId))
     { Print("Chain #", g_currentSignal.chainId, " already has an order"); return false; }
Step 4 — ICT_TradeManager.mqh: stamp the chain id into the order comment
In ExecuteTrade, in BOTH the market path and the pending path, replace:

   request.comment = InpTradeComment;
with:

   request.comment = InpTradeComment + "#" + IntegerToString(g_currentSignal.chainId);
Note: MT5 order comments are capped (~31 chars) and brokers may trim. Keep InpTradeComment short so #<id> survives. If your broker strips comments, fall back to encoding the chain id in the LAST DIGITS of a per-chain magic instead (e.g. InpMagicNumber*100 + (chainId % 100)) and match on magic ranges in HasOrderForChain.
PART B — Single Direction Resolver (#3b): kill the four duplicate fallbacks
Step 5 — ICT_SMEngine.mqh: define the single resolver
Place ABOVE GenerateSMTradeSignal (if the Entry-Mode version already exists, keep just one copy):

//+------------------------------------------------------------------+
//| Single source of truth for a chain's TRADE direction.            |
//| EntryMode override wins; else resolvedEntryDir; else direction.  |
//+------------------------------------------------------------------+
ENUM_TRADE_DIRECTION SM_ResolveTradeDirection(const SSMInstance &inst)
  {
   switch(InpSM_EntryDir)
     {
      case SM_EDIR_BUY:  return DIR_BULLISH;
      case SM_EDIR_SELL: return DIR_BEARISH;
      default:
        {
         ENUM_TRADE_DIRECTION d = inst.resolvedEntryDir;
         if(d == DIR_NONE) d = inst.direction;
         return d;
        }
     }
  }

Include-order note: SM_ResolveTradeDirection lives in ICT_SMEngine.mqh, but NAR_* lives in ICT_NarrativeGate.mqh, which is included by the detectors BEFORE the engine. To avoid an "undeclared identifier" in 6b/6c, move SM_ResolveTradeDirection into ICT_SMDetectors.mqh (or a shared header both see) so the gate can call it. Simplest: put it right after SM_GetStageDirection in ICT_SMDetectors.mqh.
Why these two are safe together
Part A only ADDS a rejection path; it can never cause an extra order, only prevent duplicates. Worst case if the comment gets trimmed by a broker: guard silently allows (fails open) — which is exactly today's behavior, so no regression. Use the magic-range fallback for strict brokers.
Part B is behavior-preserving for every existing preset (the resolver returns the same value the inline code did) EXCEPT when InpSM_EntryDir forces BUY/SELL — which is the intended Entry-Mode behavior.
